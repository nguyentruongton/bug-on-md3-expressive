# Releasing Guide

This guide details the release process for the `@bug-on` monorepo packages. 
We use [Changesets](https://github.com/changesets/changesets) for version management and **NPM Trusted Publishing (OIDC)** via GitHub Actions for secure, automated package deployment.

## Overview

The CI/CD pipeline automates publishing versions to the NPM Registry without needing to manage static `NPM_TOKEN` secrets. This guarantees a secure supply chain, with build provenance attached to our releases.

## 1. Setting Up NPM Trusted Publishing (OIDC)

Since we use NPM's OIDC support, our GitHub Actions workflow is natively trusted by NPM to publish on behalf of our organization.

**If you add a NEW package to the monorepo in the future**, follow these steps to link it:

1.  Navigate to the NPM settings page for your package context: [NPM Settings - @bug-on Packages](https://www.npmjs.com/settings/bug-on/packages).
2.  Select your new package and go to the **Settings** tab.
3.  Click on **Publishing Access** and scroll to **Trusted Publishers (OIDC)**.
4.  Configure the rule:
    *   **Publisher:** GitHub Actions
    *   **Organization or user:** `nguyentruongton`
    *   **Repository:** `bug-on-md3-expressive`
    *   **Workflow filename:** `release.yml`
5.  Save the setup.

All current packages (`@bug-on/md3-react`, `@bug-on/md3-tailwind`, `@bug-on/md3-tokens`) are already verified and set up.

## 2. Creating a Release

Whenever you complete a feature or fix a bug and want to deploy it as a new version:

### Step 2.1: Generate a Changeset
Run the following command at the root of the project:

```bash
pnpm changeset
```

1.  **Select Packages:** Use the arrow keys and spacebar to select which packages need a version bump.
2.  **Choose Bump Type:**
    *   `patch`: Bug fixes (e.g., `0.1.0` -> `0.1.1`).
    *   `minor`: New features, backwards compatible (e.g., `0.1.0` -> `0.2.0`).
    *   `major`: Breaking changes (e.g., `0.1.0` -> `1.0.0`).
3.  **Provide a Summary:** Enter a clear, user-facing summary of the changes.

A new Markdown file will be generated in the `.changeset/` folder.

### Step 2.2: Commit the Changeset
Commit the generated `.changeset/*.md` file to your branch natively.

```bash
git add .changeset/
git commit -m "chore: add changeset for my awesome feature"
git push
```

### Step 2.3: The "Version Packages" PR Loop
1.  Once your code is merged into the `main` branch, the `Release` GitHub Action (`release.yml`) will run automatically.
2.  Since there are new changesets, the Action will create or update a special Pull Request titled **"chore(release): version packages"**.
3.  This PR aggregates all unreleased changesets and prepares the `CHANGELOG.md` and `package.json` version bumps.

## 3. Publishing to NPM

1.  Review the **"chore(release): version packages"** Pull Request.
2.  Once you merge that PR into `main`, the `Release` Action will trigger again.
3.  This time, noticing that versions have been bumped and there are no unreleased `.changeset` files, the Action will automatically execute `pnpm run release` (`changeset publish`).
4.  The workflow requests an OIDC token and publishes the packages securely to NPM with full **provenance** tracking.

## Summary Checklist

- [ ] I ran `pnpm changeset` and committed the result.
- [ ] My PR was merged to `main`.
- [ ] I merged the automated "Version Packages" PR generated by Changesets.
- [ ] The GitHub Action successfully pushed to NPM.
